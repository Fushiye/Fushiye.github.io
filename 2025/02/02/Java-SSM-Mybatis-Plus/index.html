    简介

    MyBatis-Plus（简称MP）是一个MyBatis的增强工具，旨在简化开发、提高效率。它在不改变MyBatis原有功能的基础上，提供了更多的便捷特性，使得数据库操作更加简便和高效。以下是关于MyBatis-Plus的一些关键点介绍：

    核心特性：

    1. 无侵入性：完全兼容原生MyBatis，无需对现有项目进行大规模修改即可集成使用。
    2. 通用CRUD接口：提供了一套通用的Mapper接口和Service接口，支持基本的增删改查操作，极大地减少了重复代码的编写。
    3. 条件构造器：内置了强大的条件构造器Wrapper，支持链式调用，可以灵活构建查询条件，满足复杂的查询需求。
    4. 分页插件：内置了高效的分页插件，只需简单配置即可实现分页功能，减少手动编写SQL语句的工作量。
    5. 性能分析插件：帮助开发者分析SQL执行效率，找出潜在的性能瓶颈。
    6. 代码生成器：可以根据数据库表自动生成实体类、Mapper接口、Service接口及其实现类等，大大提升了开发速度。

    MyBatis-Plus适用于各种规模的企业级应用开发，特别是在需要快速迭代和频繁变更业务逻辑的情况下尤为适用。它能够显著降低开发成本，提高开发效率，特别适合那些对数据库操作有较高要求的应用程序。

    总结：

    • 自动生成单表的CRUD功能
    • 提供丰富的条件拼接方式
    • 全自动ORM类型持久层框架

    条件构造器使用

    MyBatis-Plus 提供了强大的条件构造器 Wrapper，用于构建动态查询条件。最常用的 Wrapper 实现是 QueryWrapper 和 UpdateWrapper。这些构造器可以帮助避免手动编写复杂的 SQL 语句，并且使代码更加简洁和易读.

    类结构：

    Wrapper ： 条件构造抽象类，最顶端父类

    • AbstractWrapper ： 用于查询条件封装，生成 sql 的 where 条件
      • QueryWrapper ： 查询/删除条件封装。常用
      • UpdateWrapper ： 修改条件封装。常用
      • AbstractLambdaWrapper ： 使用Lambda 语法
        • LambdaQueryWrapper ：用于Lambda语法使用的查询Wrapper。常用
        • LambdaUpdateWrapper ： Lambda 更新封装Wrappe。常用

    lambda表达式回顾

    1. 基本语法

      • (参数类型 参数名) -> { 方法体 }
      • 示例：(int a, int b) -> { return a + b; }

    2. 简化语法

      • 如果只有一个参数，可以省略括号：
        • a -> { return a * 2; }
      • 如果没有参数，使用空括号：
        • () -> { return "Hello"; }
      • 如果方法体只有一行代码，可以省略大括号和 return 关键字：
        • (a, b) -> a + b

    3. 函数式接口

      • Lambda 表达式只能用于实现函数式接口（即仅包含一个抽象方法的接口）。
      • 常见的函数式接口包括：
        • Runnable: () -> System.out.println("Running")
        • Comparator<T>: (a, b) -> a.compareTo(b)
        • Function<T, R>: x -> x * 2
        • Predicate<T>: s -> s.isEmpty()
        • Consumer<T>: t -> System.out.println(t)
        • Supplier<T>: () -> new User()

    4. 参数类型推断

      • 编译器可以根据上下文自动推断参数类型，因此通常可以省略参数类型：
        • (a, b) -> a + b 可以简写为 (a, b) -> a + b （假设 a 和 b 的类型已知）

    5. 变量捕获

      • Lambda 表达式可以访问外部的局部变量，但这些变量必须是final或effectively final（实际上不可变）：
        • String prefix = "Hello, ";
Consumer<String> greet = name -> System.out.println(prefix + name);

    6. 流操作中的应用

      • 在 Java Stream API 中，Lambda 表达式常用于定义中间操作和终端操作：
        • 过滤元素：list.stream().filter(x -> x > 10).forEach(System.out::println);
        • 映射元素：list.stream().map(String::toUpperCase).collect(Collectors.toList());
        • 聚合操作：int sum = list.stream().mapToInt(Integer::intValue).sum();

    组合条件(and/or)与高级用法

    组合条件

    主动调用or()表示前后两个条件使用or连接，不主动调用默认使用and连接

    操作类型 方法签名 示例代码 描述

    组合 OR 条件 or(Consumer<Param> consumer) queryWrapper.or(wrapper -> wrapper.eq("status", 1).or().eq("status", 2)); 组合多个 OR 条件。

    雪花算法

    雪花算法 是由 Twitter 开发的一种分布式唯一ID生成算法，能够在分布式系统中高效地生成64位的唯一ID。这个算法生成的ID是一个64位的整数，通常以字符串形式表示。

    雪花算法生成的ID由以下几个部分组成：

    • 符号位（1位）：始终为0，表示正数。
    • 时间戳（41位）：记录自定义纪元（epoch）以来的时间戳，单位通常是毫秒。可以支持大约69年的时间。
    • 工作机器ID（10位）：包括数据中心ID和工作节点ID，用于区分不同的机器。通常分为5位数据中心ID和5位工作节点ID。
    • 序列号（12位）：在同一毫秒内生成的多个ID通过序列号进行区分。12位可以支持每毫秒生成4096个不同的ID。

    在 MyBatis-Plus 中，IdType.ASSIGN_ID 的具体实现位于

    com.baomidou.mybatisplus.core.incrementer.DefaultIdentifierGenerator 类中

    @TableField注解

    @TableField注解添加在实体类非主键属性上，用于映射实体类中的字段到数据库表中的列，还可以指定字段的一些特殊行为（如是否插入、是否更新等）。如果实体类属性名与数据库表中字段名不一致时使用该注解进行映射。

    常用属性：

    • value: 数据库表中的列名（可选，默认为字段名）。
    • exist: 是否存在该字段（默认为 true）。当此属性仅为实体类属性而不是数据库字段时，使用。
    • insertStrategy: 插入策略（如 FieldStrategy.IGNORED 忽略等）。
    • updateStrategy: 更新策略（如 FieldStrategy.IGNORED 忽略等）。

    逻辑删除

    逻辑删除，可以方便地实现对数据库记录的逻辑删除而不是物理删除。逻辑删除是指通过更改记录的状态或添加标记字段来模拟删除操作，从而保留了删除前的数据，便于后续的数据分析和恢复。

    • 物理删除：真实删除，将对应数据从数据库中删除，之后查询不到此条被删除的数据
    • 逻辑删除：假删除，将对应数据中代表是否被删除字段的状态修改为“被删除状态”，之后在数据库中仍旧能看到此条数据记录

    实现方式：

    步骤：

    1. 表中添加逻辑删除字段，可以是布尔、整数、枚举类型。若为整数类型，默认 1 为逻辑删除 0 为未删除。

    2. 实体类添加逻辑删除属性。

    3. 指定逻辑删除字段和属性值。两种方式

      • 单一指定：在实体类中使用@TableLoginc注解标明字段。
      • 全局指定：在配置文件中使用logic-delete-field:配置项

    4. 此后按原方法执行正常CRUD操作，Mybatis-Plus会自动判断逻辑删除还是修改，将使用删除语句改为修改逻辑字段的语句。

    乐观锁

    在并发编程和数据库管理中，锁机制用于确保数据的一致性和完整性。乐观锁（Optimistic Locking）和悲观锁（Pessimistic Locking）是两种常见的并发控制策略。它们各自有不同的适用场景和优缺点。

    悲观锁适用场景

    • 高冲突场景：例如，多个事务频繁修改同一组数据。
    • 短事务：事务执行时间较短，锁定资源的时间也较短。

    乐观锁适用场景

    • 低冲突场景：例如，大多数情况下只有一个事务在修改数据。

    • 长事务：事务执行时间较长，锁定资源会影响其他事务的执行。

    悲观锁（Pessimistic Locking）

    • 悲观锁 假设冲突会发生，并且在进行任何操作之前先锁定资源。这意味着在读取或写入数据时，其他事务将被阻塞，直到当前事务完成并释放锁。
    • 悲观锁通常通过数据库管理系统（DBMS）的锁机制来实现，如行级锁、表级锁等。

    工作原理

    1. 获取锁：当一个事务需要访问某个资源时，首先尝试获取该资源的锁。
    2. 执行操作：在持有锁期间，事务可以安全地读取或修改数据。
    3. 释放锁：操作完成后，事务释放锁，允许其他事务访问该资源。

    乐观锁（Optimistic Locking）

    • 乐观锁 假设冲突很少发生，在进行操作时不会锁定资源。它依赖于版本号或时间戳来检测冲突。只有在提交时才会检查是否有冲突发生，如果有冲突，则回滚事务。
    • 乐观锁通常通过应用程序逻辑来实现，而不是依赖数据库的锁机制。通过版本号或时间戳来检测冲突，提高了并发性能，但在冲突频繁时可能导致大量重试。

    工作原理

    1. 读取数据：事务读取数据时，同时获取当前版本号或时间戳。
    2. 执行操作：事务对数据进行修改。
    3. 提交事务：在提交时，检查版本号或时间戳是否与读取时一致。如果一致，则更新成功；如果不一致，则说明有其他事务已经修改了数据，当前事务需要回滚或重试。

    Mybatis-Plus实现乐观锁

    步骤：

    1、添加Mybatis版本号更新拦截器（插件）（类似上文分页插件的添加方式，在主程序类中添加）
