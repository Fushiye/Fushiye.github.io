    介绍

    Docker是一种用于开发、部署和运行应用程序的开放源代码平台，它通过使用容器化技术使软件能够在任何环境中一致地运行。容器将代码及其所有依赖项打包在一起，确保应用从一个计算环境迁移到另一个计算环境时能够可靠运行。Docker的主要优点包括提高开发效率、简化配置过程、确保一致性以及提升资源利用率。

    举例来说，假设你正在开发一个Web应用程序，该程序在你的本地开发环境中运行良好。然而，当你尝试将这个应用程序部署到生产服务器上时，可能会遇到各种问题，比如：

    • 环境差异：生产服务器可能安装了不同版本的操作系统、数据库或其他关键软件，这可能导致应用程序无法正常运行。
    • 依赖管理：不同的项目可能需要不同版本的库或框架，而这些依赖关系之间可能存在冲突。
    • 团队协作问题：如果团队成员的工作环境不一致（操作系统、工具版本等），可能会导致“在我机器上能跑”的问题。

    使用Docker的好处：

    1. 一致性：通过Docker，你可以为你的应用创建一个包含所有依赖项的镜像，这样无论是在开发、测试还是生产环境中，应用都能以相同的方式运行。
    2. 轻量级虚拟化：与传统的虚拟机相比，Docker容器更加轻量，启动速度更快，并且更有效地利用系统资源。
    3. 快速部署：Docker允许你快速搭建服务环境，极大地加快了应用的部署流程。
    4. 易于扩展和维护：可以轻松地将应用部署到多个服务器上，或者根据需要调整规模，同时保持环境的一致性。

    容器化技术

    特点：轻量、快速、隔离、跨平台、高密度（部署）

    基本语法：具体可以通过 docker run --help 查看

    • IMAGE: 要运行的镜像名称或ID。
    • COMMAND: 容器启动后要执行的命令（可选）。
    • ARG: 传递给命令的参数（可选）。
    • [OPTIONS]: 各种选项来控制容器的行为。

    常用参数：

    1. 运行模式

      • -d, --detach: 在后台运行容器。

      • --rm: 容器退出时自动删除容器。

    2. 端口映射

      • -p, --publish list: 将容器的端口映射到主机的端口。

        这个例子将容器的80端口映射到主机的8080端口。

    3. 卷挂载

      • -v, --volume list: 将主机目录挂载到容器中。

        这个例子将主机的 /host/path 目录挂载到容器的 /container/path 目录。

    4. 环境变量

      • -e, --env list: 设置环境变量。

    5. 名称与网络

      • --name string: 给容器指定一个名称。

      • --network string: 连接到指定的网络。

    6. 重启策略

      • --restart string: 定义容器的重启策略（no, on-failure, always, unless-stopped）。

    7. 交互模式

      • -i, --interactive: 保持STDIN打开，即使没有附加也保持容器运行。

      • -t, --tty: 分配一个伪TTY（终端）。

        这个例子会启动一个Ubuntu容器，并进入交互式bash shell。

    8. 配置资源限制

      • -m, --memory bytes: 设置容器内存限制。

      • --cpus decimal: 设置容器可用的CPU数量。

    网络

    主要网络模式

    • Bridge模式：这是Docker默认的网络模式。Docker daemon会在主机上创建一个名为docker0的虚拟网桥，并为每个容器分配独立的网络命名空间和IP地址。容器间的通信可以通过这个网桥实现，而对外部的访问则需要通过端口映射来完成。

    • Host模式：容器直接使用宿主机的网络栈，不提供网络隔离。这种方式减少了网络延迟并提高了性能，但牺牲了安全性。

    • Overlay模式：用于跨主机的容器间通信。它依赖于Docker Swarm或Kubernetes等集群管理工具，能够在不同物理或虚拟机上的容器之间建立一个虚拟网络，使它们可以像在同一台机器上一样相互通信。

    • None模式：容器没有任何网络配置，适合那些不需要网络功能的应用。

    工作原理

    当启动一个容器时，如果没有指定网络模式，默认会使用bridge模式。Docker daemon会自动为该容器分配一个虚拟网卡，并将其连接到docker0网桥上。由此容器就拥有了一个独立的内部IP地址，可以通过docker inspect containerName查看容器对应的内部IP。

    但是，通过IP地址进行容器间通信可能会因为容器的开启关闭而导致的IP频繁发生变化带来不便，因此在多容器环境中，Docker提供内置的DNS服务器，使得容器可以通过名称而不是IP地址相互发现和通信。默认的docker0网桥不支持域名访问，因此需要通过docker network create netName来创建一个自定义网络，并在容器启动时使用--network netName指定使用的网络，使用同一个网络的容器可以直接通过容器名:端口访问。

    跨主机通信：对于需要跨多个主机通信的场景，可以使用Overlay网络。首先，在各主机上部署Docker Engine，并初始化Swarm集群，然后创建一个Overlay网络，最后将容器加入到这个网络中。

    Docker Compose 是一个用于定义和运行多容器 Docker 应用程序的工具。通过一个 docker-compose.yml 文件，你可以配置应用程序的服务、网络和卷等，然后使用一条命令来启动、停止和管理这些服务。

    3. 管理和维护服务：使用其他 Docker Compose 命令来管理服务。

    分层机制

    Docker镜像采用分层存储机制，每个镜像由一系列只读的层组成，每层代表镜像的一部分如操作系统、应用程序及其依赖等。当创建容器时，会在镜像的最上层添加一个可写的层。

    • 基础层：通常是一个操作系统环境，例如Ubuntu或Alpine Linux。
    • 中间层：包含应用运行所需的各种库和依赖项，每执行一条RUN指令会形成一个新的层。
    • 顶层（可写层）：当基于镜像启动容器时添加，用于存储容器运行时产生的数据变化，如文件修改、新增或删除。

    例如，当我们在官方Nginx镜像的基础上创建新的Nginx镜像时，实际上是在利用同一个基础层。这就是Docker镜像分层机制的一大优势，具体来说：

    共享基础层：

    • 官方Nginx镜像：这个镜像是由Nginx团队或Docker官方维护的，它包含了运行Nginx所需的基本环境和配置。这一部分构成了镜像的基础层。

    • 基于官方Nginx镜像的新镜像：当使用FROM nginx指令开始编写Dockerfile并在此基础上添加自定义配置、静态文件或其他依赖时，实际上是在官方Nginx镜像的顶部添加新的层。原有的基础层（即官方Nginx镜像的部分）保持不变。

    实际效果：

    • 存储效率：由于基础层是只读的并且可以被多个镜像共享，因此无论在该基础镜像上构建了多少个不同的镜像，这些镜像都会共享同一份基础层的数据，从而节省了大量的存储空间。

    • 传输效率：当将基于相同基础镜像构建的新镜像推送到Docker仓库或者拉取到本地时，如果目标系统中已经存在了相同的基础层，那么只需要下载新增加的层即可，这大大加快了镜像的传输速度。
