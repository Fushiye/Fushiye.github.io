介绍

消息队列

消息队列是一种跨进程或跨服务器的通信方式，用于在不同的应用程序组件之间异步传递消息。它允许应用程序通过发送和接收消息来进行解耦，提高系统的灵活性、可扩展性和可靠性。

  • 关键点：
    • 异步通信：消息的生产者（发送方）和消费者（接收方）不需要同时在线。
    • 解耦：生产者和消费者互不依赖，仅通过消息队列进行交互。
    • 缓冲：可以在高负载时起到缓冲作用，平滑系统峰值流量。
    • 可靠性：支持持久化存储，确保消息不会丢失。

RabbitMQ

RabbitMQ是一个实现高级消息队列协议（AMQP）的消息代理软件，广泛用于构建可靠的消息队列系统。

  • 关键点：
    • 开源：基于Erlang语言开发，完全开源且社区活跃。
    • 多协议支持：除了AMQP外，还支持MQTT、STOMP等多种协议。
    • 灵活的路由：提供了多种消息路由机制，如直接、主题、头信息和扇出交换机等。
    • 易于使用：提供丰富的客户端库，支持多种编程语言和平台。
    • 高可用性：支持集群部署，提高系统的容错能力和处理能力。

相关概念

交换机

在RabbitMQ中，交换机（Exchange）是消息首先到达的地方，然后根据特定的规则将消息路由到一个或多个队列。RabbitMQ支持几种不同类型的交换机，每种类型决定了消息如何被路由。

1. 直接交换机（Direct Exchange）

直接交换机是最基础的一种交换机类型，它通过消息的路由键（Routing Key）与队列绑定时指定的路由键进行精确匹配来路由消息。只有当消息的路由键与某个队列绑定时设置的路由键完全一致时，该消息才会被发送到对应的队列。

  • 使用场景：适用于需要根据明确的消息类别进行处理的情况。
2. 主题交换机（Topic Exchange）

主题交换机允许更灵活的路由策略，它基于模式匹配的方式进行路由键的匹配。这里的路由键可以包含两个特殊字符：*（匹配一个单词）和#（匹配零个或多个单词）。这使得消息可以根据部分匹配规则被路由到相应的队列。

  • 使用场景：适用于需要对消息进行分类处理，尤其是当消息类别存在层次结构或模式时。
3. 扇形交换机（Fanout Exchange）

扇形交换机不考虑路由键，它简单地将消息广播给所有绑定了该交换机的队列。这意味着只要队列绑定了这个交换机，就一定能收到该交换机的所有消息，而无需关心消息的路由键是什么。

  • 使用场景：适用于需要向多个消费者同时发送相同消息的场景，如日志记录、实时更新通知等。

除了上述三种主要的交换机类型，RabbitMQ还支持其他类型的交换机，例如头部交换机（Headers Exchange），它允许根据消息头属性而非路由键来进行匹配。选择哪种类型的交换机取决于应用的具体需求和消息传递的逻辑。

通道和队列

  • 通道（Channel）：可以将通道理解为客户端与消息队列服务器之间的通信管道。一个TCP连接上可以创建多个通道，每个通道就像是在这条TCP连接上的独立“子连接”，它们共享同一个TCP连接但各自操作互不干扰。使用多通道可以在降低资源消耗的同时实现多任务处理。
  • 队列（Queue）：队列是存储消息的地方。生产者将消息发送到交换器（Exchange），然后根据路由键（Routing Key）和绑定规则（Bindings），这些消息会被转发到一个或多个队列中。消费者则从队列中读取消息进行处理。、

工作模式

模式名称 描述 特点
简单模式 包含一个生产者、一个队列和一个消费者。 最基本的消息传递方式，适用于简单的场景。
工作队列模式 多个消费者共享来自同一个队列的任务。 适合用于处理耗时任务，能够平衡工作负载。
发布/订阅模式 消息被广播给所有监听该交换器的消费者。 所有订阅者都会收到相同的消息，实现消息的广泛传播。
路由模式 根据特定的路由键（routing key）发送消息到指定队列。 允许根据关键字精确控制消息流向，提高了消息传递的灵活性。
主题模式 基于路由模式，但支持使用通配符进行更灵活的消息路由。 提供了更复杂的匹配规则，使得消息投递更加精准和灵活。
头部模式 类似于路由模式，但使用消息头部属性而不是路由键进行匹配。 不依赖于路由键，而是通过头部信息来决定消息的去向。
RPC模式 利用RabbitMQ作为远程过程调用机制，客户端请求服务端并等待响应。 支持远程过程调用，允许分布式系统中的进程相互调用对方的服务。

工作队列（WorkQueues）

工作队列模式（Work Queues Mode），也被称为任务队列模式，是一种用于在多个工作者（消费者）之间分配耗时任务的模式。它旨在避免需要等待长时间运行的任务完成，而是将这些任务放入一个队列中，让多个工作者异步处理这些任务。这种模式非常适合用来处理耗时的操作，如图片处理、视频转换等。

也就是说，多个消费者监听同一个消息队列，队列中的消息会被这些消费者轮流消费（竞争关系），一条消息只会被一个消费者拿到，适用于任务较重或较多的情况。多个消费者分摊任务可以提高效率。

发布订阅（Publish/Subscribe）

简称Pub/Sub模式，是一种设计模式，它通过消息的传递机制实现了组件之间的解耦。我们这里说的发布/订阅模式，指的是基于Fanout交换机（扇形交换机）的发布订阅模式，也就是基础的发布订阅模式。在这种模式中，消息的发送者（称为发布者）不会直接将消息发送给特定的接收者（称为订阅者）。相反，发布者会向Fanout交换机发送消息，而这个交换机会无差别地将消息分发给所有与其绑定的队列。因此，即使多个队列绑定了同一个交换机，每条消息也会被复制并分别放入每个队列中。最终，每个消费者（Consumer）从其对应的队列中读取消息时，实际上是在处理原始消息的一个副本。。如果多个消费者绑定到同一个队列，则队列中的消息会被这些消费者轮流消费（竞争关系），而不是每个消费者都收到消息的副本。

路由模式（Routing)

这也是一种发布订阅模式，但是它是基于Direct交换机（直接交换机）的。在这种模式中，消息的发送者（称为发布者）会向Direct交换机发送消息，而这个交换机会根据路由键地将消息分发给与对应路由键绑定的队列。因此，只有对应路由键的队列才会收到消息，其他队列不会收到消息。最终，每个消费者（Consumer）从其对应的队列中读取消息时，实际上是在处理原始消息。如果多个消费者绑定到同一个队列，则队列中的消息同样会被这些消费者轮流消费（竞争关系），而不是每个消费者都收到消息的副本。

主体模式（Topic）

主题模式（Topic）也是一种发布订阅模式，但它基于Topic交换机（主题交换机）。在这种模式中，消息的发送者（称为发布者）会向Topic交换机发送带有特定格式路由键的消息，并且这个交换机会基于路由键中的特殊字符与绑定到队列上的模式进行模糊匹配来分发消息。只有当消息的路由键与某个队列绑定的路由键模式匹配时，该队列才会收到这条消息。而如果一个消息的路由键同时匹配多个队列，它就会被分配到所有匹配的队列。如假设一个消息路由键 apple.orange.banana， 那它即会被分配到 apple.#队列，也会被分配到*.orange.*队列。如果多个消费者绑定到同一个队列，则这些消费者之间形成竞争关系，每条消息只会被其中一个消费者处理。

两个主要通配符是：

  • * (星号) 可以替代一个单词。
  • # (井号) 可以替代零个或多个单词。

头部模式（Header）

头部模式也是一种发布订阅模式，但它不依赖于路由键（Routing Key），而是使用消息头中的属性来决定如何路由消息。在头部模式下，生产者发送消息时可以附加一系列的头信息（headers），这些头信息是以键值对的形式存在的。交换机会根据绑定队列时设置的头部参数来判断消息是否应该被路由到该队列。头部模式支持两种类型的匹配：

  • 任何（Any）：只要消息头部包含与绑定中定义的任意一个头部键值对相匹配，消息就会被路由到该队列。
  • 所有（All）：消息头部需要包含并且匹配绑定中定义的所有头部键值对，消息才会被路由到该队列。

头部模式提供了更大的灵活性，因为它允许匹配多个属性，并且可以指定这些属性是否必须满足条件。

远程过程调用模式（RPC）

RPC（远程过程调用）模式是一种允许程序通过网络调用其他地址空间（可能位于另一台计算机上）的过程或函数的编程模型，其设计目的是让这种跨网络的调用与普通的本地方法调用一样简单和直观。工作原理大致如下：客户端发起对某个远程过程的调用请求，这个请求包含了必要的参数信息，并且这些信息需要经过序列化处理以便于在网络中传输；请求到达服务器后，服务器负责反序列化接收到的数据并执行对应的过程或函数，然后将执行的结果再次序列化并通过网络返回给客户端；最后，客户端反序列化结果数据，完成整个调用过程。通过这种方式，RPC隐藏了底层网络通信的复杂性，使得分布式系统的开发变得更加容易。

消息可靠性传递

在RabbitMQ消息传递过程中，可能会出现多种故障导致消息没有被正确处理，以下是常见故障及解决思路：

  • 故障情况1：消息没有发送到消息队列

    • 解决思路A：在生产者端进行确认，具体操作中我们会分别针对交换机和队列来确认，如果没有成功发送到消息队列服务器上，那就可以尝试重新发送。
    • 解决思路B：为目标交换机指定备份交换机，当目标交换机投递失败时，把消息投递至备份交换机。

  • 故障情况2：消息队列服务器宕机导致内存中消息丢失

    • 解决思路：消息持久化到硬盘上，哪怕服务器重启也不会导致消息丢失。

  • 故障情况3：消费端宕机或抛异常导致消息没有成功被消费

    • 消费端消费消息成功，给服务器返回ACK信息，然后消息队列删除该消息。
    • 消费端消费消息失败，给服务器端返回NACK信息，同时把消息恢复为待消费的状态，这样就可以再次取回消息，重试一次（当然，这就需要消费端接口支持幂等性）。

每当有消息进入队列并准备被消费时，RabbitMQ 会为这条消息生成一个 deliveryTag。这个标签是基于当前通道的，并且在该通道内针对特定队列是递增的64位整数。消费者从队列中获取消息后，如果

  1. 处理成功：调用 basic.ack 方法，并传入正确的 deliveryTag。表示消费者已经成功处理了消息，服务器可以安全地将该消息从队列中移除。

  2. 处理失败且希望重新处理消息（例如临时性错误）：调用 basic.nack 方法，并设置 requeue 参数为 true，同时传入正确的 deliveryTag。这样会使消息被重新放入队列尾部，等待再次传递给同一个或其他消费者进行重试。

  3. 处理失败并不再处理此消息（例如永久性错误）：调用 basic.nack 或 basic.reject 方法，并设置 requeue 参数为 false，同时传入正确的 deliveryTag。这样做会告知服务器直接丢弃该消息，不再尝试重新投递。basic.nack 和 basic.reject 在这种情况下效果相同，但 basic.nack 支持批量拒绝消息。

  4. 预取消息处理超时或异常退出（未手动确认或拒绝消息）：取决于服务器端的配置（如自动确认设置）和客户端的心跳检测机制。如果设置了自动确认模式，则消息会被认为已被处理；若未设置，消息可能会根据服务质量（QoS）设置而重新分发给其他消费者。

  5. 在事务中处理消息：使用 AMQP 的事务特性，在一个事务中包含多个消息的接收和确认操作。如果事务提交成功，则所有相关消息都会被确认；如果回滚，则这些消息将被重新放入队列。这种方式提供了更严格的控制，但性能通常比非事务性操作要低。

RabbitMQ 支持一次性确认多条消息，即批量确认。可以通过设置 multiple 参数为 true 实现，这样可以提高效率，减少网络通信次数。由于 deliveryTag 在每个通道中是唯一的，所以在使用多个通道时需要注意不要混淆 deliveryTag。此外，如果连接断开且没有启用自动重连，那么所有未确认的消息将会被重新发送给其他消费者或者留在队列中等待下一次消费。

消费端限流

默认情况下，RabbitMQ会采用轮询的方式将消息分发给所有活跃的消费者。如果只有一个消费者连接到队列，并且没有配置限流，理论上这个消费者有可能接收所有的消息，导致消费者被消息淹没。因此，通常需要设置预取值（Prefetch Count）。预取值定义了未确认消息的最大数量，这些消息可以一次性发送到一个特定的消费者，从而限制了一个消费者处理的最大消息量。

修改消费者配置文件：设置配置项spring-rabbitmq-listener-simple-prefetch

死信处理

当一个消息无法被消费，就会变成死信。

死信产生有三种情况：

  • 拒绝：basicNack()或basicReject()且requeue=false.
  • 溢出：队列消息数量达到限制后，再来消息时，最早的消息会变成死信
  • 超时：消息达到超时时间未被消费

死信一般有三种处理方式

  • 丢弃：不重要的消息不做处理直接丢弃
  • 入库：写入数据库日后处理
  • 监听：进入死信队列，专门设置消费者监听死信队列并做后续处理。常用

事务消息

在RabbitMQ事务机制下，当启用事务txSelect()后使用如convertAndSend()这样的方法发送消息时，消息并不会立即发送到队列，而是暂时保存在缓存中，直到事务成功提交（通过txCommit()）。若事务提交成功，所有缓存的消息将一并发送；若事务执行了回滚（通过txRollback()），则这些消息不会被发送，并从缓存中删除。这种方式不仅确保了消息的成功发送，同时也提供了处理发送失败情况下的回滚机制，从而维护系统的数据一致性。

惰性队列

惰性队列（Lazy Queue）是 RabbitMQ 中的一种队列行为模式，它旨在优化队列性能，尤其是在处理大量消息时。默认情况下，RabbitMQ 队列在存储消息时会尝试保持一定的平衡：既能够快速地将消息传递给消费者，也要有效地使用内存和磁盘资源。然而，在某些情况下，例如当队列需要存储大量的消息而这些消息不会立即被消费者处理时，这种平衡可能并不理想。惰性队列非常适合那些预计会有大量积压消息但不需要立即处理的场景。例如，用于日志收集、事件记录等用途的消息队列，其中消息可能需要长期存储以供后续分析或审计。

惰性队列的主要特点

  • 减少内存使用：当启用惰性队列选项时，消息尽可能快地从内存转移到磁盘上。这意味着即使队列中有大量未处理的消息，它们也不会占用过多的内存资源。

  • 提高稳定性：由于减少了内存中的数据量，RabbitMQ 节点可以更稳定地运行，特别是在面对高负载或突发流量的情况下。

在声明队列时，可以通过设置队列参数来启用惰性队列。具体来说，可以通过设置 x-queue-mode 参数为 lazy 来启用此模式：

优先级队列

RabbitMQ 的优先级队列允许您根据消息的优先级来决定处理顺序。这意味着高优先级的消息可以被优先消费，而不是按照它们到达队列的顺序进行处理。这对于需要紧急处理的任务特别有用。

  1. 定义队列时设置x-max-priority参数：这是配置队列以支持优先级消息的关键步骤。该参数指定了队列中消息的最高优先级（通常为0到某个值，0代表最低优先级）。
  2. 发布消息时指定优先级：当您发布消息到一个已配置的优先级队列时，可以通过设置priority属性来指定每条消息的优先级。
