SSM整合理论

SSM框架如何整合应用？

  • 微观：将学习的Spring SpringMVC Mybatis框架应用到项目中!
    • SpringMVC框架负责控制层
    • Spring 框架负责整体和业务层的声明式事务管理
    • MyBatis框架负责数据库访问层
  • 宏观：Spring接管一切（将框架核心组件交给Spring进行IoC管理），代码更加简洁。
    • SpringMVC管理表述层、SpringMVC相关组件
    • Spring管理业务层、持久层、以及数据库相关（DataSource,MyBatis）的组件
    • 使用IoC的方式管理一切所需组件
  • 实施：通过编写配置文件，实现SpringIoC容器接管一切组件。

三层架构

层级名称 职责描述 常见技术实现
表示层Controller 处理用户交互，接收请求并返回响应（如 HTTP API、Web 页面等） Spring MVC、Thymeleaf、JSP、RESTful API
业务逻辑层
Service		 实现核心业务规则和流程，协调数据访问层的操作 Spring @Service、事务管理、业务逻辑类
数据访问层
Mapper/Repository		 负责与数据库或其他数据源交互（CRUD 操作） Spring Data JPA、MyBatis、JDBC、Hibernate

核心问题

第一问：SSM整合后需要几个IoC容器？

两个容器
本质上说，整合就是将三层架构和框架核心API组件交给SpringIoC容器管理！
一个容器可能就够了，但是我们常见的操作是创建两个IoC容器（web容器和root容器），组件分类管理！
这种做法有以下好处和目的：

  1. 分离关注点：通过初始化两个容器，可以将各个层次的关注点进行分离。这种分离使得各个层次的组件能够更好地聚焦于各自的责任和功能。
  2. 解耦合：各个层次组件分离装配不同的IoC容器，这样可以进行解耦。这种解耦合使得各个模块可以独立操作和测试，提高了代码的可维护性和可测试性。
  3. 灵活配置：通过使用两个容器，可以为每个容器提供各自的配置，以满足不同层次和组件的特定需求。每个配置文件也更加清晰和灵活。

总的来说，初始化两个容器在SSM整合中可以实现关注点分离、解耦合、灵活配置等好处。它们各自负责不同的层次和功能，并通过合适的集成方式协同工作，提供一个高效、可维护和可扩展的应用程序架构！

第二问：每个IoC容器对应哪些类型组件？

容器名 盛放组件
web容器 web相关组件（controller,springmvc核心组件）
root容器 业务和持久层相关组件（service,aop,tx,dataSource,mybatis,mapper等）

第三问：IoC容器之间关系和调用方向？

情况1：两个无关联IoC容器之间的组件无法注入！

情况2：子IoC容器可以单向的注入父IoC容器的组件！

结论：web容器是root容器的子容器，父子容器关系。

  • 父容器：root容器，盛放service、mapper、mybatis等相关组件
  • 子容器：web容器，盛放controller、web相关组件

第四问：具体多少配置类以及对应容器关系？

配置类的数量不是固定的，但是至少要两个，为了方便编写，我们可以三层架构每层对应一个配置类，分别指定两个容器加载即可！

三层架构框架

创建三层架构软件包：controller、service、mapper(repository)；在resources文件夹下建一个与mapper同级的mapper目录存放xml文件。

controller 目录：处理HTTP请求并返回响应。

  • 接收客户端的请求（如通过REST API）。
  • 调用相应的服务层方法来处理业务逻辑。
  • 将处理结果转换为适当的格式（如JSON或XML）并返回给客户端。
  • 处理异常和错误，并提供友好的错误消息。

service 目录：包含业务逻辑和处理规则。

  • 实现具体的业务逻辑。
  • 调用数据访问层（如mapper或repository）进行数据操作。
  • 对外提供清晰的接口，供控制器或其他服务调用。
  • 可以包含事务管理、缓存等高级功能。

mapper 或 repository 目录：负责与数据库交互，执行CRUD操作。

  • 定义数据访问接口，通常使用MyBatis、JPA等ORM框架实现。
  • 提供基本的增删改查（CRUD）操作。
  • 可以包含复杂的查询逻辑，如联表查询、聚合函数等。
  • 在MyBatis中，mapper通常是一个接口，Spring会自动实现它。

controller

创建ScheduleController类，添加@RestController注解以支持返回JSON数据并标记一个类为Spring MVC控制器，处理HTTP请求并将结果返回给客户端。添加@RequestMapping("schedule")注解确定其处理的请求URL

通常要创建接口和实现类，创建一个ScheduleService接口，用于声明业务层处理方法；在子目录Imp下创建对应实现类ScheduleServiceImp，并添加@service注解标识和配置服务层组件，便于Spring自动管理和依赖注入。

问题1 跨域问题解决

问题描述

由于前端部署访问端口为http://localhost:5173/ 后端为http://localhost:8080/ 会出现跨域问题：

跨域问题（Cross-Origin Resource Sharing, CORS） 是指在浏览器环境中，当一个前端应用试图从一个不同的域名、协议或端口请求资源时，浏览器出于安全考虑会阻止这种请求。这是由于浏览器的同源策略（Same-Origin Policy）所引起的。

同源策略：同源——两个 URL 的协议、域名和端口号完全相同

解决方案

在前后端交互中，解决跨域问题的主要方案包括：

  1. 服务器端配置：最常用的解决方案之一是通过在后端服务器上设置HTTP响应头来允许特定或所有来源的请求。例如，在Java应用中使用@CrossOrigin注解或者在Spring Boot中全局配置CORS。（本次实践中，通过在ScheduleController类中添加注解@CrossOrigin解决跨域问题

  2. JSONP（JSON with Padding）：一种通过<script>标签绕过浏览器同源策略的方法，但仅支持GET请求，现在已较少使用。

  3. 代理服务器：前端应用可以配置一个代理服务器，将API请求转发到不同源的服务器上，以此避免跨域问题。这种方式常用于开发环境。

  4. CORS Filter：对于不支持CORS配置的服务器，可以通过实现一个过滤器（Filter）来添加必要的CORS响应头。

  5. WebSocket：虽然主要用于全双工通信，但在某些情况下也可以用来规避跨域限制。

常见错误

错误1 数据库连接错误：

[ERROR] [Druid-ConnectionPool-Create-897634805] [com.alibaba.druid.pool.DruidDataSource] [create connection SQLException, url: jdbc:mysql://localhost:3306/test, errorCode 1045, state 28000] java.sql.SQLException: Access denied for user ‘${db.user}‘@’localhost’ (using password: YES)

排查过程：

  1. 确定jdbc.properties中账户密码正确，不能有空格、引号
  2. 确定DataSource配置文件正确扫描到jdbc.properties
  3. 确定DataSource配置文件中@Value()注解使用了${}，且其中名称必须与jdbc.properties一致！！！！鼠标悬停可见是否正确读取。确保@Value()注解括号内除了${参数名}外没有多打字符！！！
